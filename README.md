# PitchIQ — Premier League, decoded.

A premium, data-driven web application and encyclopedia dedicated to the English Premier League. **PitchIQ** delivers comprehensive historical data, live match updates, club profiles, and an advanced player comparison engine — including **cross-era head-to-heads**: search any player from any era (1992–present) — by name, nickname, or initials (cr7, kdb) — and pick each at any season, or their whole career via "All seasons". Modern seasons (2017–present) include **expected goals (xG) + expected assists (xA)** on profiles and head-to-heads (TASK-M20).

## ✨ What PitchIQ demonstrates

- **Modern Next.js 15 App Router** — server components, Route Handlers, streaming `<Suspense>`, and file-based metadata (dynamic `next/og` cards, PWA manifest, SVG favicon).
- **Type-safe data layer** — every read is parsed and validated with **Zod**; a committed-JSON snapshot architecture (no runtime database) keeps the app fast, free to host, and offline-testable.
- **Automated data pipeline** — an **external pipeline** rebuilds the dataset daily and opens an **auto-PR** to this repo when anything changes. The footer shows a live **"Data updated X ago"** stamp (from `data/_meta.json`) so the refresh cadence is visible (TASK-M22).
- **Depth of data** — **34 seasons** (1992-93 → 2025-26 — the complete Premier League history) with **stable cross-season player ids**, half-time scores, referees, match attendance + stadium, clean sheets + saves, a dedicated leaderboards page, synthesized recent form, **real starting-XI lineups + match events** (goals/cards/subs) for every season (SDP API 2008-2025, legacy PL API 1992-2007), and **player stats + leaderboards for every season** (derived from lineups/events for 2010-2016, and sourced from the official Premier League stats API for 1992-2009 and 2025-26).
- **Provable-fact "Did You Know?" engine** — a deterministic (no-LLM) rule library (**26 rules** across player/team/league scopes) that surfaces only facts it can re-derive from the committed data, each with a `verify` closure. Spans event-mined player insights (clubs scored against, favourite opponent, hat-tricks), per-season player angles (xG over/under-performance, clean-sheet rate, transfer impact), team patterns (unbeaten-at-home, comeback wins, managerial churn, yo-yo relegations), and league records (biggest crowds, the safety line, 100-point seasons, giant-killers).
- **Interactive historic map** — a `/map` page plots all 51 clubs on an **accurate England & Wales map** (real ONS region boundaries, projected) with a **1992→2025 timeline slider** that lights up each season's top flight and re-skins the whole page through the eras, plus clickable region modals (clubs + combined PL titles).
- **Quality engineering** — **1425 unit/component tests** (Vitest) + **Playwright E2E** + **visual-regression** coverage, **Sentry** observability, and CI gates on every PR.
- **Design system** — **Tailwind CSS v4** + **Shadcn UI**, semantic color tokens, light/dark themes, and a custom PL-purple brand identity.

- **Deep Club Profiles (`/teams/[id]`):** Dynamic Server-Side Rendered (SSR) routes detailing club histories, stadium details, current squads, and recent form metrics.

## 🛠️ Tech Stack & Engineering Decisions

- **Framework:** `Next.js 15 (App Router)` & `React 19` — Server Components and Route Handlers handle all data reads server-side; Client Components are reserved for interactive surfaces like the Comparison engine and live filtering.
- **Language:** `TypeScript` (strict) — Strictly typed interfaces mapping the committed data snapshots to guarantee absolute compile-time safety.
- **Data Source:** Committed **JSON snapshots** under `data/` (standings / teams / players / fixtures / leaderboards / lineups / events, plus a cross-season search index). These are produced and refreshed daily by an **external data pipeline** that opens data-only PRs to this repo; the app itself has no data-fetching/scraper code — it only reads the committed snapshots. No API key needed.
- **Server-Side Data Layer:** `src/data/loaders.ts` — server-only async loaders read committed JSON snapshots via `readFile` and validate against Zod schemas in `src/data/schemas.ts`. Each feature's `src/features/*/api.ts` is a thin adapter that calls the loaders and reshapes the snapshot's flat shape into the consumer-facing types (legacy from Phase 2–4; reshape kept to minimise component churn during the migration). Returns `null` on ENOENT / parse error / schema violation so pages degrade to empty states rather than crash.
- **Client-Side Data:** `TanStack Query (v5)` — Manages cache lifetimes (`staleTime`), background refetching, and optimistic updates **only** for client-interactive features (search filters, Comparison engine refetches, live updates). It never duplicates SSR data fetching.
- **URL State:** `nuqs` — Type-safe, sharable URL search-params drive the Player Comparison selections (`?a=<id>&b=<id>`) and filter state, replacing a global client store. Copy-pastable comparison links come for free.
- **Styling & Components:** `Tailwind CSS v4` + `Shadcn UI` — Utility-first styling paired with accessible, unstyled primitives. Skeleton loaders optimize CLS/LCP on hydration boundaries.
- **Motion:** Hybrid — CSS/Tailwind keyframes + the native View Transitions API for most animation, with `Motion` (framer-motion's successor) reserved for orchestrated sequences and **lazy-loaded only** (zero First-Load cost on pages that don't use it). Shared duration/easing tokens live in `globals.css`, mirrored by `src/utils/motion.ts` with a parity test; everything is `prefers-reduced-motion`-gated per the policy in [`docs/motion.md`](docs/motion.md). A game-style **neon boot loader** (SSR-painted pure CSS, once per session, era-aware, scroll-locked) plays on first load, every internal navigation runs a **zoom-fade View Transition** (wrapped at the app's central `Link`; instant fallback where unsupported), page content — cards, tiles, list rows, sections — **soft-rises into view** with a staggered, IntersectionObserver-driven scroll reveal that never hides content from no-JS or reduced-motion visitors, and every interactive surface speaks one **"neon glow" micro-interaction language** (an era-accent halo on hover, gentle press compress, glowing tabs and command palette — magenta in the modern era, cyan in the golden, claret in the retro).
- **Testing:** `Vitest` + `@testing-library/react` (unit/component, happy-dom) and `Playwright` (E2E) — Vitest covers hooks, utilities, and isolated components; Playwright exercises SSR-rendered routes like `/teams/[id]` and the end-to-end Comparison flow.
- **Observability:** A structured JSON `logger` in `src/utils/logger.ts` used by loaders + Route Handlers + error boundaries. `logger.warn` / `logger.error` forward to Sentry in production (`logger.info` / `debug` stay console-only to keep event volume manageable on the free tier). Root-render errors caught by `src/app/global-error.tsx`; Route Handler exceptions captured via `instrumentation.ts#onRequestError = Sentry.captureRequestError`.

## 🚀 Deployment

Production is hosted on **Vercel** with the GitHub integration: every push to `main` deploys to production, every PR gets a preview URL posted as a status check.

### One-time setup (Vercel dashboard)

1. **Import the repo.** Vercel dashboard → `Add New… → Project` → select `AliEmad0/pitchiq` → `Import`. Vercel auto-detects Next.js; leave the build/install commands at the defaults (`pnpm build` / `pnpm install`).
2. **Add env vars.** Project → `Settings → Environment Variables`. Paste the values from your local `.env.local`. At minimum:
   - `REVALIDATE_SECRET` (Production, Preview) — gates `GET /api/admin/revalidate?tag=…&secret=…`. Generate with `openssl rand -hex 32`. Without it, the endpoint returns 401 for every request.
   - `NEXT_PUBLIC_SITE_URL` (Production) — optional. Canonical absolute origin used by `metadataBase` in `src/app/layout.tsx` (drives OG/Twitter image URLs). When unset, the resolver falls back to `https://${VERCEL_URL}` (auto-injected per deploy) and finally `http://localhost:3000` in dev.
   - `NEXT_PUBLIC_SENTRY_DSN` (Production, Preview, Development) — the Sentry DSN (browser-visible by design — that's how Sentry's Public Key model works). Without it, `Sentry.init` no-ops and no events leave the app. With it, `logger.warn`/`logger.error` forward to the Issues feed in production and Route Handler exceptions are captured via `instrumentation.ts#onRequestError`.
   - `SENTRY_AUTH_TOKEN` + `SENTRY_ORG` + `SENTRY_PROJECT` (Production, Preview) — optional, gate source-map upload during production builds so Sentry stack traces are unminified. Without them the build still succeeds; the bundler plugin just skips the upload step.
   - (Note: no sports-data API key is needed — the app reads committed data snapshots. If a legacy `API_KEY` / `API_BASE_URL` is still set in your project, deleting it is harmless housekeeping.)
3. **Lock production to `main`.** Project → `Settings → Git`. Set `Production Branch` to `main`. The default already deploys previews on every PR — leave that on.
4. **Disable Vercel's auto-deploy on direct pushes to feature branches** (optional but recommended). Same `Git` settings panel → `Ignored Build Step`: set to `bash -c 'if [ "$VERCEL_GIT_COMMIT_REF" = "main" ] || [ -n "$VERCEL_GIT_PULL_REQUEST_ID" ]; then exit 1; else exit 0; fi'`. This builds only for `main` pushes and PRs, not for every direct branch push.

- **Preview:** open a PR → Vercel bot posts a `Visit Preview` link as a status check within ~90s.
- **Production:** merge to `main` → check the Deployments tab; the deploy should reach `Ready` within ~3 minutes.
- **Smoke test:** on either environment, `curl $DEPLOY_URL/api/health` should return HTTP 200 with `{ status: "ok", commit, uptime, data: { lastRefresh, datasets }, ts }` — confirms the committed data is reachable from the deployed function.
